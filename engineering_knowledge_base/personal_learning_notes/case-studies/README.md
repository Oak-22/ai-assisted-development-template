# Case Studies

This README defines the purpose of the case studies section of the
engineering knowledge base.

Use case studies for deeper reflections on technical problems,
tradeoffs, and debugging paths.

## Recommended Case Study Structure

1. Situation: system context, constraints, and stakeholders.
2. Problem: what was failing or blocked.
3. Investigation: hypotheses, diagnostics, and evidence gathered.
4. Decision: selected solution and why alternatives were rejected.
5. Outcome: measurable impact and residual risk.
6. Follow-up: prevention work and monitoring improvements.

- `debugging-cache-invalidation-causing-stale-pricing.md`

## Quality Bar

Strong case studies are specific, evidence-backed, and honest about
tradeoffs and missed assumptions.
